# Walkthrough Conversion Style Guide

This guide outlines exactly how to convert raw markdown source files from the original guide into the styled Astro components used in this project. 

When an AI or developer converts a new walkthrough page, they MUST adhere to the following mapping rules to maintain consistent design and functionality (such as the interactive sidebar TOC and the lightbox overlay).

## 1. Page Wrapper & Frontmatter
Every walkthrough page must import and use the `WalkthroughLayout` component. The local TOC is generated by passing a `sections` array to the layout.

```astro
---
import WalkthroughLayout from "../../layouts/WalkthroughLayout.astro";

// Automatically mapped from the <h2> headers in the source
const sections = [
  { id: "intro", label: "Intro" },
  { id: "fiorwoods", label: "Fiorwoods" },
  // ...
];
---

<WalkthroughLayout
  title="Page Title (e.g., World Exploration)"
  sections={sections}
>
  <!-- Content goes here -->
</WalkthroughLayout>
```

## 2. Structural Mapping (Sections)
Each top-level header (`##`) from the markdown must become a distinct HTML section.

**Markdown Source:**
```markdown
## Fiorwoods
Here is the text...
```

**Astro Conversion:**
```html
<section id="fiorwoods" class="walkthrough-section">
  <h2 class="section-title">Fiorwoods</h2>
  <div class="step-content">
    <p>Here is the text...</p>
  </div>
</section>
```
**Rules:**
- `id` must match the id provided in the `sections` array for the TOC to work.
- All paragraphs and media within a section MUST be wrapped inside `<div class="step-content">`.

## 3. Media Elements (Images)
Images should be converted based on whether they are solitary or side-by-side in the source.

**Single Image Mapping:**
```html
<div class="media-container wide">
  <a href="HIGH_RES_URL" target="_blank">
    <img src="THUMBNAIL_URL" alt="Alt text" />
  </a>
</div>
```

**Double Image Mapping (Side-by-side):**
```html
<div class="media-grid">
  <a href="HIGH_RES_URL_1" target="_blank">
    <img src="THUMBNAIL_URL_1" alt="Alt 1" />
  </a>
  <a href="HIGH_RES_URL_2" target="_blank">
    <img src="THUMBNAIL_URL_2" alt="Alt 2" />
  </a>
</div>
```
*(Note: Do not worry about lightbox functionality, as `WalkthroughLayout.astro` handles URL transformation and overlays automatically for all `<img>` tags).*

## 4. Callouts, Tips, and Notes
Any paragraph starting with "NOTE:" or appearing as a blockquote should be styled as a tip card.

**Markdown Source:**
```markdown
*NOTE:* You should take lunch to the hubby...
```

**Astro Conversion:**
```html
<div class="tip-card">
  <p><strong>NOTE:</strong> You should take lunch to the hubby...</p>
</div>
```

## 5. Boss Fights and Encounters
Major combat sections or Class Emblem challenges should be highlighted in a boss-fight card.

**Markdown Source:**
```markdown
### CLASS EMBLEM CHALLENGE: BANDIT
We are to fight 4 Terror Terriers...
```

**Astro Conversion:**
```html
<div class="boss-fight-card">
  <h4 class="boss-title">CLASS EMBLEM CHALLENGE: BANDIT</h4>
  <p>We are to fight 4 <strong>Terror Terriers</strong>...</p>
</div>
```

## 6. Next Steps / Links Out
The very end of a walkthrough page typically links to the next guide or main quest. Use the `next-steps-card` for this.

**Markdown Source:**
```markdown
Next: [Main Quest - Searching for a Princess](/walkthrough/searching-for-a-princess)
```

**Astro Conversion:**
```html
<div class="next-steps-card">
  <a href="/walkthrough/searching-for-a-princess">Main Quest - Searching for a Princess</a>
</div>
```

## 7. Global Content Introduction
If there is introductory text *before* the first `<h2>` header, it should be wrapped in a `.story-blurb` directly underneath the `<WalkthroughLayout>` opening tag.

```html
<div class="story-blurb">
  <p>Welcome to one of the biggest pages...</p>
</div>

## 8. Astro Compiler Syntax Requirements
Astro uses Esbuild under the hood, so standard JS module syntax and JSX rules apply inside `.astro` files.

**Frontmatter Import Order:**
- All `import` statements **MUST** be placed at the very top of the frontmatter section.
- You cannot define any constants (like `const baseUrl = ...`) prior to your imports. Placing variable declarations before `import` statements will break the Esbuild parser and throw misleading "Unterminated string literal" errors further down the file.

```astro
---
// CORRECT
import WalkthroughLayout from "../../layouts/WalkthroughLayout.astro";
const baseUrl = import.meta.env.BASE_URL;
---
```

**JSX Text Escaping:**
- Bare ampersand (`&`) characters inside JSX text nodes (e.g. `Sword & Shield`) can sometimes trigger parser errors during production builds. 
- ALWAYS convert bare ampersands in text into HTML entities (`&amp;`) when hardcoding them inside the Astro template. (e.g. `Sword &amp; Shield`).
